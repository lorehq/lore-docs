title: "Hooks"
---

The `lore` CLI binary handles all hook events. Platform configs point to `lore hook <name>`, which reads JSON from stdin and writes JSON or plain text to stdout.

## Platform hook configuration

Each platform projector generates the appropriate hook config:

| Platform | Config file | Events |
|----------|------------|--------|
| Claude Code | `.claude/settings.json` | `PreToolUse`, `PostToolUse`, `UserPromptSubmit` |
| GitHub Copilot | `.github/hooks/lore.json` | `preToolUse`, `postToolUse`, `userPromptSubmitted` |
| Cursor | `.cursor/hooks.json` | `preToolUse`, `postToolUse`, `beforeSubmitPrompt` |
| Gemini CLI | `.gemini/settings.json` | `BeforeTool`, `AfterTool`, `BeforeAgent` |
| Windsurf | `.windsurf/hooks.json` | `pre_write_code`, `post_write_code`, `pre_run_command`, `pre_read_code`, `pre_user_prompt` |
| OpenCode | JS plugins (`.opencode/plugins/`) | `tool.execute.before`, `tool.execute.after` (no prompt-submit equivalent) |

All configs are generated by `lore generate` and point to `lore hook <name>`.

## Hook subcommands

### lore hook pre-tool-use

| | |
|---|---|
| **Platform events** | `PreToolUse`, `preToolUse`, `BeforeTool`, `pre_write_code`, `pre_run_command` |
| **Input** | JSON with `tool_name` and `tool_input` (file paths, arguments) |
| **Output** | JSON deny decision or plain text guidance |
| **Profile gating** | Guards run in all profiles; search-nudge skipped in `minimal` |

Three guards run in sequence:

**Harness guard** — gates writes to the global directory (`~/.lore/`). If the resolved path starts with the global directory, prompts the operator for approval (`ask` decision). Project-level `.lore/` paths are allowed.

**Memory guard** — blocks all access to `MEMORY.md` at the project root. Read attempts get a redirect message pointing to `.lore/MEMORY.md`. Write attempts get routing guidance: snags to fieldnotes, knowledge to `~/.lore/MEMORY/DATABANK/`, temporary to `.lore/MEMORY.md`.

**Search nudge** — detects Read/Glob on indexed paths (`docs/`, `.lore/AGENTIC/SKILLS/`, `.lore/AGENTIC/RULES/`) and nudges the agent to use semantic search first. Non-blocking — always allows the tool to proceed. Messages prefixed with `[LORE-MEMORY]`.

### lore hook post-tool-use

| | |
|---|---|
| **Platform events** | `PostToolUse`, `postToolUse`, `AfterTool`, `post_write_code` |
| **Input** | JSON with `tool_name`, `tool_input`, exit status |
| **Output** | Plain text capture nudges |
| **Profile gating** | Skipped in `minimal` |

Adaptive memory nudges after tool use. Tracks consecutive bash command count in `.git/lore-bash-count-<hash>.json`:

- Read-only tools (`Read`, `Grep`, `Glob`): silent
- Knowledge-path writes: silent (already capturing)
- First bash in a sequence: memory nudge
- Bash count hits `nudgeThreshold` (default 15): nudge to capture findings
- Bash count hits `warnThreshold` (default 30, repeats every multiple): stronger nudge to pause and capture
All messages use green `[■ LORE-MEMORY]` tags.

### lore hook prompt-submit

| | |
|---|---|
| **Platform events** | `UserPromptSubmit`, `userPromptSubmitted`, `beforeSubmitPrompt`, `BeforeAgent`, `pre_user_prompt` |
| **Input** | JSON with `user_input` text |
| **Output** | Plain text ambiguity warning (if matches found) |
| **Profile gating** | Skipped in `minimal` |

Scans for ambiguous input patterns (relative time, vague quantities, open-ended scope). Appends a yellow `[AMBIGUITY]` warning if matches are found.

## Hook logging

Enabled by `LORE_HOOK_LOG=1`. Writes JSONL to `.git/lore-hook-events.jsonl` (falls back to OS temp dir if `.git/` does not exist). Each line:

```json
{ "ts": "2025-03-01T12:00:00Z", "hook": "pre-tool-use", "event": "PreToolUse", "output_size": 142 }
```

## Deny mechanism

The `lore hook` subcommand outputs a single JSON format for all platforms:

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",
    "permissionDecisionReason": "..."
  }
}
```

For approval gates (harness guard), `permissionDecision` is `"ask"` instead of `"deny"`. For non-blocking context (search nudge), the response uses `additionalContext` instead of a permission decision.

## Color system

Hook output uses ANSI colors to signal context. Each message is prefixed with a colored tag in the format `\x1b[NNm[■ LORE-TAG]\x1b[0m`:

| Color | ANSI code | Tags | Hooks |
|-------|-----------|------|-------|
| Green | `\x1b[92m` | `LORE-MEMORY` | post-tool-use nudges |
| Yellow | `\x1b[93m` | `AMBIGUITY` | prompt-submit (ambiguity scanner) |

Pre-tool-use guards (harness guard, memory guard, search nudge) output uncolored tags (`[HARNESS-GUARD]`, `[MEMORY-GUARD]`, `[LORE-MEMORY]`).
